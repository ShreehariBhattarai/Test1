Direct Lake fallback: why your "instant" report just got slow

Query folding is what makes Power Query fast: instead of pulling every row into memory and transforming it locally, the engine translates your steps into a single query that runs at the source. The moment folding breaks, every step after that point runs locally, row by row. Refreshes that took thirty seconds start taking thirty minutes.

How to check

Right-click any step in the Applied Steps pane. If "View Native Query" is available, that step still folds. The first step where that option disappears is where folding broke — everything from there onward runs locally.

Common folding breakers

  • Adding a custom column with a function the source can't translate
  • Changing data types in a way the source connector doesn't support pushing down
  • Merging or appending queries from genuinely different sources
  • Using Table.Buffer or other functions that deliberately force local evaluation

Three habits

Do your filters and column removals early, before any custom logic — fewer rows and columns folded early means less work either way. Push transformations that have a native SQL equivalent ahead of anything bespoke. And when you do need a folding-breaking step, put it as late as possible in the query, so as little work as possible happens after the break.
