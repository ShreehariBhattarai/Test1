Direct Lake vs. Import vs. DirectQuery: Choosing the Right Storage Mode

Shreehari Bhattarai June 14, 2026 9 min read Fabric

On this page

  • How Direct Lake works
  • When to use Import
  • When to use DirectQuery
  • Decision framework
  • Common pitfalls
  • FAQ

One of the first architectural decisions in any Fabric semantic model is the storage mode, and it's a decision that's expensive to get wrong after the fact. I've rebuilt models from Import to Direct Lake mid-project because nobody weighed freshness requirements against dataset size upfront. Here's the framework I use now.

The three modes

Microsoft Fabric semantic models support three storage modes, and each makes a different tradeoff between query speed, data freshness, and infrastructure cost:

  • Import loads data into VertiPaq, the in-memory columnar engine that also powers classic Power BI. Fast queries, but data is only as fresh as the last refresh.
  • DirectQuery sends native queries to the source system for every visual interaction. Always fresh, but query performance depends entirely on the source.
  • Direct Lake is Fabric-specific: it reads Delta Parquet files straight from OneLake without a separate import step, combining near-VertiPaq speed with near-real-time freshness.

How Direct Lake actually works

Direct Lake doesn't copy data into the semantic model the way Import does. Instead, it memory-maps Delta Parquet row groups from OneLake directly into the analytical engine on first query, then keeps "hot" columns cached for subsequent queries. When the underlying Lakehouse or Warehouse table changes, Fabric can trigger an automatic framing operation that picks up the new Parquet files without a traditional refresh.

The practical effect: your Lakehouse can update every few minutes via a pipeline, and your Power BI report reflects that without you managing a separate refresh schedule for the semantic model.

The catch is fallback. If a query pattern isn't supported in Direct Lake (certain complex DAX, or a table exceeding guardrail limits for your capacity SKU), Fabric silently falls back to DirectQuery against the Warehouse SQL endpoint, which is slower. Watch for this in Performance Analyzer if a report that was fast suddenly isn't.

When to use Import mode

Import mode still wins in a few specific situations:

  • The dataset is small enough that VertiPaq's compression makes the whole model trivially fast, and freshness requirements are daily or slower.
  • The model needs calculated tables, calculated columns, or DAX patterns that aren't yet supported against Direct Lake sources.
  • Data doesn't originate in a Fabric Lakehouse or Warehouse at all — for example, a mix of SharePoint lists and small Excel files.

Import is also the safer default when you're prototyping. It's the mode with the fewest surprises, because the entire dataset genuinely lives in memory once loaded.

When to use DirectQuery

DirectQuery makes sense when freshness truly cannot wait for any caching layer — think trading desks or operational dashboards tied to a transactional system — or when the source dataset is too large to import economically, such as a multi-terabyte warehouse fact table that a handful of analysts query occasionally.

The tradeoff is real: every slicer click issues a new query against the source, so report performance is bounded by your SQL endpoint's concurrency and indexing, not by Power BI.

A decision framework

decision-logic.txt
Is the source a Fabric Lakehouse or Warehouse?
  No  -> Import (or DirectQuery if source demands live data)
  Yes -> Does the report need near-real-time freshness (minutes, not hours)?
           No  -> Import is simplest; use it unless the model is huge
           Yes -> Does the DAX/model use only Direct-Lake-supported patterns?
                    Yes -> Direct Lake
                    No  -> DirectQuery against the Warehouse SQL endpoint,
                           or restructure the model to drop the blocking pattern

Common pitfalls I've seen

  1. Assuming Direct Lake is "free" performance. The first query against a cold column still pays a read cost from OneLake. Warm the model with a scheduled query if first-load latency matters to users.
  2. Ignoring capacity guardrails. Each Fabric capacity SKU has row group and table size limits for Direct Lake before automatic fallback to DirectQuery kicks in. Check these against your fact table sizes before committing to the mode.
  3. Building calculated columns into a Direct Lake model. These force a fallback for the whole table. Push that logic upstream into the Lakehouse or Warehouse instead.

FAQ

Is Direct Lake always faster than Import mode?

Not always. Import's in-memory VertiPaq engine can outperform Direct Lake on smaller models with heavy DAX, because Direct Lake has to read Parquet row groups on first access. Direct Lake wins mainly on freshness and scale, not raw query speed on small datasets.

Can I mix storage modes in one semantic model?

Yes — this is a composite model. You can combine Direct Lake or Import tables with DirectQuery tables, which is useful when most data can be modeled in Fabric but one source must remain live.

Does Direct Lake require Microsoft Fabric?

Yes. Direct Lake reads Delta Parquet tables directly from OneLake, so it only applies to Fabric Lakehouses and Warehouses, not arbitrary external databases.
